## 本节目标

- 掌握从零开发一个 Skill 的完整 7 步路线
- 理解每一步的核心动作和产出
- 学会用 brainstorming Skill 做需求收敛，避免后期返工
- 学会用 skill-creator 评估质量，确保 Skill 规范可用
- 通过真实案例（社交平台书签导出）理解全流程

---

## 开发路线图总览

如果你找遍了 3 个渠道都没有满足需求的 Skill，或者现有的只能解决部分问题，那就自己造。

整个开发流程分 7 步：

| 步骤 | 核心动作 | 产出 |
|------|----------|------|
| Step 1 | 先搜索，不要重复造轮子 | 确认没有现成方案 |
| Step 2 | 有参考就 fork + 改造 | 基于已有 Skill 快速起步 |
| Step 3 | 没有参考，先把需求说清楚 | 明确的需求描述 |
| Step 4 | 用 brainstorming Skill 做设计 | 收敛后的设计文档 |
| Step 5 | 开始实现 | 可运行的 Skill 代码 |
| Step 6 | 用 skill-creator 评估质量 | 质量检查报告 + 修复 |
| Step 7 | 实测验证 | 确认 Skill 在真实场景下可用 |

下面逐步展开。

---

## Step 1：先搜索，不要重复造轮子

回到 [第 01 章](/skill/01-find/) 的 3 个渠道，用关键词搜一遍：

- **skill.sh**：`npx skills find [关键词]`
- **clawhub**：按分类浏览
- **GitHub**：搜 `claude skill` 或 `awesome-claude-skills`

哪怕找到一个功能接近的，也能省你大量时间。不要觉得"差不多"就不看了，很多时候你想做的事情别人已经做了 80%。

---

## Step 2：有参考就 fork + 改造

如果找到了一个能解决部分问题的 Skill，直接 fork 下来改。

改比从零写快得多。你可以：

- fork 仓库到自己的 GitHub
- clone 到本地
- 修改 SKILL.md 中的指令逻辑和 frontmatter
- 调整实现代码

这一步的关键是**不要追求完美**，先跑通再优化。

---

## Step 3：没有参考，先把需求说清楚

这一步最关键。不是上来就写代码，而是先把需求阐述清楚。

以一个真实案例说明：我每天刷社交平台会收藏大量帖子，但书签积累多了根本看不过来。我需要的是：

- 把书签批量导出为 Markdown
- 用模型做分类筛选、生成摘要
- 这样我就能快速检索，而不是一条条翻

需求说清楚了，方案就自然浮现了。

:::tip[需求描述的标准]
一个好的需求描述应该包含三个要素：**当前痛点**（现在什么情况让你难受）、**期望结果**（你想要达到什么效果）、**使用场景**（谁在什么时候用）。不需要写得很正式，但这三点要覆盖到。
:::

---

## Step 4：用 brainstorming Skill 做设计（重点）

这一步很多人会跳过，但它直接决定了你后面返不返工。

### 什么是 brainstorming Skill

Superpowers 提供了一个 `brainstorming` skill（安装方式见 [第 03 章 Superpowers 部分](/skill/03-recommended/)）。你只需要用自然语言描述需求，它会反问你一系列问题来逐步收敛，把模糊想法变成明确设计。

### 需求收敛的关键问题

以社交平台书签导出 Skill 为例，brainstorming 过程中需要回答的核心问题：

**1. 做 PoC 还是正式版？**

PoC（概念验证）只需要跑通核心流程，不用考虑异常处理、性能优化。正式版则需要完整的错误处理、日志、配置项。先搞清楚你现在处于哪个阶段，避免过度设计或设计不足。

**2. 导出范围是什么？**

全量导出还是按时间段？最近 10 条还是最近 50 条？有没有分类筛选的需求？范围不同，实现复杂度差异很大。

**3. 输出什么格式？**

单个大文件还是每条一个文件？Markdown 还是 JSON？文件名怎么命名？目录结构怎么组织？这些看似细节的问题，如果不提前定好，后面改起来代价很高。

**4. 重复运行怎么处理？**

第二次跑的时候，已导出的内容要不要跳过？是覆盖还是追加？这就是幂等性的设计。如果不考虑这个，你的 Skill 只能跑一次，第二次跑就会出问题。

**5. 执行顺序是什么？**

先登录还是先检查环境？先导出数据还是先创建目录？把执行步骤排清楚，实现的时候才不会乱。

一轮问答下来，模糊需求变成了明确的设计文档和实施计划。**先设计再动手，这是工程思维的基本功。**

---

## Step 5：开始实现

到了这一步，懂技术的可以自己选型实现，不懂技术的可以直接把设计文档丢给高级模型，让它帮你写。

这就是 **vibe coding** 的价值所在——你不需要会写代码，但你需要会描述需求。前面 4 步做扎实了，这一步就是把设计文档交给模型执行。

以社交平台书签导出 Skill 为例，最终的实现思路：

- 读取本地浏览器中你自己的登录状态
- 通过官方接口按页读取你的个人书签数据
- 每条书签转换为独立 Markdown 文件

这些不需要你自己想，把需求和设计文档交给模型，它会帮你搞定实现。

:::note[关于 vibe coding]
vibe coding 不是让你完全不懂技术，而是把精力从"怎么写代码"转移到"怎么描述清楚我要什么"。前面 Step 3 和 Step 4 的需求收敛，就是 vibe coding 能不能成功的关键。
:::

## Step 6：用 skill-creator 评估质量

写完之后不要急着用，先让专业工具帮你检查一遍。

### 安装 skill-creator

Anthropic 官方提供了一个 `skill-creator` skill，专门用来评估你写的 Skill 的质量。

安装方式：直接在 Claude Code 对话中输入：

```text
安装更新 https://github.com/anthropics/skills/tree/main/skills/skill-creator
```

也可以手动 clone 后放到 `~/.claude/skills/` 目录。

### 评估维度

skill-creator 会从以下 4 个维度帮你检查：

| 维度 | 检查内容 |
|------|----------|
| 结构 | frontmatter 格式是否正确、SKILL.md 文件是否完整 |
| 描述 | `description` 字段是否清晰、能否让 AI 正确匹配触发 |
| 触发条件 | 触发方式是否明确（主动调用 vs 自动匹配）、是否和已有 Skill 冲突 |
| 安全边界 | 是否有越权操作、敏感信息处理是否安全、依赖是否声明 |

跑完评估后，它会告诉你哪里需要改。按建议修复，直到评估通过。

---

## Step 7：实测验证

评估通过后，真实跑一遍。

以社交平台书签导出 Skill 为例，启动后它会交互式地问你：

- 导出全部 / 50 条 / 10 条？
- 是否下载视频 / 图片等资源？
- 要不要生成总结？

回答完这些问题后，Skill 开始执行，逐条读取书签、转换为 Markdown、生成分类摘要。

### 验收三连

实测的时候，用这三个问题做验收：

1. **核心流程跑通了吗？** -- 主路径是否能从头到尾走完，输出结果是否符合预期
2. **重复运行正常吗？** -- 第二次跑的时候，是否正确跳过已导出内容（幂等性验证）
3. **异常情况处理了吗？** -- 网络断开、登录态过期、数据为空等情况是否有合理的错误提示

三个都通过，这个 Skill 就可以正式投入使用了。

## frontmatter 规范说明

每个 Skill 的 `SKILL.md` 文件都以 frontmatter 开头，其中两个字段最重要：

```yaml
---
name: my-skill-name
description: 一句话说明这个 skill 干什么
---
```

### name 字段

- 决定了斜杠命令的名称：`/my-skill-name`
- 必须是小写字母 + 连字符，不要用空格或中文
- 取名要有辨识度，别叫 `my-skill` 这种通用名

### description 字段

- **最关键的字段**，直接决定 AI 能不能正确触发你的 Skill
- 对于自动触发型 Skill，AI 会根据 description 判断当前任务是否匹配
- 写得太模糊会导致误触发或不触发
- 建议格式：`[动作] + [对象] + [场景]`，例如 `"批量导出社交平台书签为 Markdown 文件，支持去重和摘要生成"`

:::caution[常见错误]
description 写成 `"一个有用的工具"` 这种说了等于没说的内容，AI 根本无法判断什么时候该触发它。务必写清楚这个 Skill 具体做什么。
:::
